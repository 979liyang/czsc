# Research: 麒麟分析涨停跌停与买卖点时间节点

**Feature**: 002-analyze-points-table  
**Phase**: 0

## 1. 涨停/跌停判定规则

**Decision**: 采用与 `bar_zdt_V230331` 一致的逻辑：对当前周期下每根 K 线，若 `close == high` 且 `close >= 前一根 close` 则记为涨停；若 `close == low` 且 `close <= 前一根 close` 则记为跌停。在分析区间内对 `bars_raw` 全量扫描，收集满足条件的 K 线 `dt` 作为时间节点。

**Rationale**: 项目内已有 `czsc/signals/bar.py::bar_zdt_V230331` 使用该规则，策略层多处引用（如 `涨跌停V230331_涨停`/`跌停`），保持同一套定义便于一致性与复用。

**Alternatives considered**:
- 按 A 股涨跌幅 10%/20% 阈值：需前收、标的类型（ST/科创板等），且仅在日线有意义；当前需求为「不同周期」均要时间节点，K 线级 close/high/low 规则适用于任意周期。
- 后续若产品要求严格按交易所涨跌停定义，可在本逻辑之上增加「仅日线 + 涨跌幅阈值」的配置扩展。

---

## 2. 第一/二/三类买卖点识别方式

**Decision**: 复用项目内已有缠论信号，在历史 K 线上按时间顺序回放 CZSC（或等价地逐 bar 更新并取信号），收集信号触发的时刻作为对应买卖点时间节点。

- **第一类买点/卖点**：使用 `zdy_macd_bs1_V230422`，其 `v1` 为「看多」视为一买、「看空」视为一卖；时间节点取该信号触发时当前 K 线的 `dt`（即 D1 对应的 bar.dt）。
- **第二类买点/卖点**：使用 `cxt_second_bs_V230320`，`v1` 为「二买」「二卖」时收集 `dt`。
- **第三类买点/卖点**：使用 `cxt_third_bs_V230319`（或 V230318），`v1` 为「三买」「三卖」时收集 `dt`。

实现上：在 CZSC 构造完成后，对 `bars_raw` 按时间顺序做「回放」（或从空 CZSC 逐 bar update），在每根 K 线处计算上述信号，若 v1 匹配则将该 bar 的 `dt` 加入对应类型列表。信号参数采用默认（如 di=1、均线周期等），与策略中常用配置一致即可。

**Rationale**: 规格假定「以项目内已有或即将引入的缠论实现为准」；现有信号已覆盖一二三类买卖点且被策略使用，直接复用可避免重复实现且保证与回测/实盘逻辑一致。

**Alternatives considered**:
- 仅用笔/分型结构定义一买一卖（如底分型=一买）：与现有策略使用的「MACD 背驰」一买不一致，易造成同一页面展示与策略信号不一致。
- 新增一套专用买卖点识别：重复建设，且需额外维护两套定义。

## 3. 时间节点格式与粒度

**Decision**: 序列化时按「当前分析周期」决定时间节点格式：日线、周线、月线仅输出日期（YYYY-MM-DD）；15/30/60 分钟等输出日期+时间（YYYY-MM-DD HH:mm 或 ISO 8601 含时分），以便同一日内区分不同 K 线。后端在序列化层根据 `freq` 选择格式；前端展示可直接使用该字符串，无需再根据周期分支。

**Rationale**: 满足规格 FR-005 与 SC-004：时间节点粒度与周期一致，且便于人工核对与导出。

**Alternatives considered**:
- 统一返回 ISO 8601 含时分秒：日线会多出 00:00:00，展示略冗余；可在前端再按周期格式化，但增加前端逻辑，故优先在后端按周期输出。

---

## 4. 与现有分析 API 的集成方式

**Decision**: 不新增 API 端点；在现有麒麟分析接口（如 `POST /analysis/czsc`）的响应体中扩展字段，增加八类事件的列表（如 `limit_up_events`, `limit_down_events`, `buy1_events`, `buy2_events`, `buy3_events`, `sell1_events`, `sell2_events`, `sell3_events`），每项为 `{ dt, ... }` 的数组。请求参数不变（symbol, freq, sdt, edt），后端在现有分析流程中调用 CZSC 新能力并写入 result。

**Rationale**: 规格要求「麒麟分析页面」展示表格，即一次分析请求即拿到全部数据；现有前端已按「单次分析单周期」使用该 API，扩展响应字段成本最小且不破坏兼容性（未使用的客户端可忽略新字段）。

---

## 5. 小结

| 主题           | 结论 |
|----------------|------|
| 涨停/跌停      | 使用 bar_zdt_V230331 逻辑，全量扫描 bars_raw 收集 dt |
| 一二三类买卖点 | 使用 MACD BS1 / BS2 / BS3 辅助信号，回放 K 线收集 v1 匹配的 dt |
| 时间格式       | 按 freq：日/周/月仅日期，分钟周期日期+时间 |
| API            | 扩展现有分析响应，不新增端点 |

以上均无需 NEEDS CLARIFICATION，可直接进入 Phase 1 设计与契约编写。
